# 03. Спецификация первой рабочей версии (MVP)

**Продукт:** ProjectMind (AI Marketing System)  
**Версия документа:** 0.1  
**Статус:** проектирование первой рабочей версии  
**Основание:** `00_PROJECT_RULES.md`, `01_PROJECT_CHARTER.md`

---

## 1. Назначение документа

Этот документ фиксирует границы первой **рабочей** версии продукта — не прототипа и не учебного проекта. Цель MVP: через несколько дней передать первому реальному клиенту инструмент, который позволяет собрать структурированную диагностику проекта и получить от ИИ полезный анализ для дальнейшей маркетинговой работы.

Принцип продукта из устава:

> Сначала понимание проекта → потом стратегия → потом упаковка → потом сайт → потом реклама.

MVP покрывает этап **«понимание проекта»** и даёт основу для **«стратегии и упаковки»**. Создание сайта и запуск рекламы в первую версию не входят.

| `01_PRODUCT_VISION.md` | **пуст** |

### 2.2. Целевой пользователь MVP

**Основной пользователь:** маркетолог, который ведёт проекты клиентов.

**Вторичный участник сценария:** клиент маркетолога — заполняет диагностику самостоятельно, без участия специалиста в момент ответов.

### 2.3. Критерии успеха MVP (из устава)

Первая версия считается успешной, если:

1. клиент может **самостоятельно** заполнить диагностику;
2. вся информация **сохраняется в одном месте**;
3. ИИ формирует **полезный и структурированный** анализ;
4. специалисту **не требуется повторно задавать** основные вопросы клиенту;
5. результаты анализа **можно использовать** для создания лендинга, рекламной кампании и дальнейшей работы.

## 3. Функции, обязательные для первой версии

### 3.1. Управление проектами (минимальное)

| ID | Функция | Описание |
|----|---------|----------|
| F-01 | Создание проекта | Маркетолог создаёт проект с базовыми данными: название, клиент, краткое описание сферы/услуги |
| F-02 | Список проектов | Просмотр всех проектов маркетолога с индикацией статуса диагностики |
| F-03 | Карточка проекта | Единая точка входа в проект: статус, ссылка на диагностику, ответы, результаты анализа |
| F-04 | Архивирование / закрытие проекта | Возможность пометить проект как завершённый (без удаления данных) |

### 3.2. Диагностика

| ID | Функция | Описание |
|----|---------|----------|
| F-05 | Структурированный опросник | Фиксированный набор блоков вопросов, покрывающий ключевые темы: продукт/услуга, клиент, аудитория, конкуренты, цели, текущее состояние, ограничения |
| F-06 | Пошаговое заполнение | Клиент проходит диагностику блоками; видит прогресс; может сохранить и вернуться позже |
| F-07 | Доступ по ссылке | Маркетолог получает уникальную ссылку для клиента — клиенту не нужна регистрация в системе |
| F-08 | Автосохранение ответов | Каждый ответ сохраняется сразу; потеря данных при обрыве сессии недопустима |
| F-09 | Завершение диагностики | Явное действие «Завершить» с подтверждением; после завершения редактирование клиентом блокируется (маркетолог может разблокировать) |
| F-10 | Уведомление о завершении | Маркетолог видит, что клиент завершил диагностику (статус в карточке проекта) |

### 3.3. Хранение и просмотр собранной информации

| ID | Функция | Описание |
|----|---------|----------|
| F-11 | Единое хранилище ответов | Все ответы клиента доступны маркетологу в структурированном виде по блокам |
| F-12 | Просмотр «сырых» ответов | Маркетолог может читать оригинальные формулировки клиента без искажений |

### 3.4. ИИ-анализ

| ID | Функция | Описание |
|----|---------|----------|
| F-13 | Автоматический запуск анализа | После завершения диагностики система автоматически формирует анализ на основе ответов |
| F-14 | Структурированный отчёт по разделам | Результат разбит на логические блоки (см. раздел 7) |
| F-15 | Список недостающей информации | ИИ явно указывает, каких данных не хватает или где ответы слишком общие |
| F-16 | Повторная генерация | Маркетолог может запросить пересборку анализа (например, после уточнения ответов) |

### 3.5. Работа с результатами

| ID | Функция | Описание |
|----|---------|----------|
| F-17 | Просмотр результатов в интерфейсе | Маркетолог читает анализ внутри системы, раздел за разделом |
| F-18 | Экспорт итогового отчёта в PDF | Скачиваемый документ для передачи клиенту или использования offline |
| F-19 | Копирование фрагментов | Возможность скопировать текст отдельного раздела (для вставки в документы, брифы, ТЗ) |

### 3.6. Доступ маркетолога

| ID | Функция | Описание |
|----|---------|----------|
| F-20 | Вход в систему | Маркетолог авторизуется и видит только свои проекты |
| F-21 | Базовый профиль | Имя, контакт — для подписи в PDF-отчёте |

## 4. Функции, которые нельзя делать в первой версии

### 4.1. Явно исключено уставом проекта

- CRM (воронки, сделки, этапы продаж)
- Биллинг и оплата
- Подписки и тарифные планы
- Управление сотрудниками и ролями внутри агентства
- Финансовый учёт
- Управление рекламными кабинетами
- Автоматический запуск рекламы
- Автоматические рассылки (email, мессенджеры)
- Сложная аналитика (дашборды, метрики, A/B-тесты)

### 4.2. Дополнительно исключено для MVP (чтобы уложиться в срок и не размыть фокус)

| Функция | Причина исключения |
|---------|-------------------|
| Редактор лендингов / конструктор сайтов | Принцип «сначала понимание» — сайт после диагностики, не внутри неё |
| Публикация и хостинг лендингов | За рамками MVP |
| Генерация и публикация рекламных объявлений | За рамками MVP |
| Интеграции с внешними сервисами (CRM, Notion, Google Docs, Telegram-бот) | Увеличивает scope и риски перед первым клиентом |
| Мультипользовательский доступ к одному проекту (несколько маркетологов) | Достаточно одного специалиста на проект |
| Кастомизация опросника маркетологом | Фиксированный опросник снижает риск ошибок; кастомизация — v2 |
| Версионирование ответов и diff изменений | Достаточно текущего состояния + возможность разблокировки |
| Чат с ИИ внутри проекта | Достаточно структурированного отчёта |
| Мобильное приложение | Адаптивная веб-версия достаточна |
| Мультиязычность интерфейса | MVP на русском языке |
| White-label / брендирование под агентство | v2+ |
| Клиентский кабинет с историей всех проектов | Клиент работает только по ссылке на текущую диагностику |

## 5. Минимальный пользовательский сценарий (Happy Path)

### Участники

- **Мария** — маркетолог, основной пользователь системы
- **Анна** — клиент Марии, владелица онлайн-школы

### Сценарий: от создания проекта до готового отчёта

```
Мария                          Система                         Анна
  |                               |                              |
  |-- создаёт проект ------------>|                              |
  |<-- карточка проекта ----------|                              |
  |-- копирует ссылку ----------->|                              |
  |-- отправляет ссылку Анне -----|------------------------------>|
  |                               |<-- открывает диагностику ----|
  |                               |<-- отвечает блок 1 ----------|
  |                               |   (автосохранение)           |
  |                               |<-- сохраняет, уходит ----------|
  |                               |   (через 2 дня)              |
  |                               |<-- продолжает блоки 2–5 ------|
  |                               |<-- завершает диагностику -----|
  |                               |-- запускает ИИ-анализ ------>|
  |<-- статус «Анализ готов» -----|                              |
  |-- открывает результаты ------>|                              |
  |-- читает разделы, правит ------|  (при необходимости          |
  |   недостающие ответы          |   разблокирует для Анны)     |
  |-- скачивает PDF -------------->|                              |
  |-- использует отчёт для -------|                              |
     брифа на лендинг и рекламу   |                              |
```

### Обязательные ветки сценария

| Ветка | Поведение системы |
|-------|-------------------|
| Клиент бросил на середине | Прогресс сохранён; ссылка работает; маркетолог видит «В процессе, 40%» |
| Клиент дал поверхностные ответы | ИИ формирует анализ + блок «Недостающая информация» с конкретными вопросами |
| Маркетолог разблокировал диагностику | Клиент может дополнить ответы; маркетолог перезапускает анализ |
| Повторный вход маркетолога | Видит актуальный статус всех проектов |

### Что сценарий **не** обязан покрывать в MVP

- Совместная работа двух маркетологов
- Оплата доступа клиентом
- Автоматическая отправка PDF клиенту по email (маркетолог отправляет вручную)

## 8. Что получает пользователь после заполнения диагностики

### 8.1. Маркетолог получает

1. **Полный набор структурированных ответов клиента** — без необходимости перечитывать переписку или проводить повторное интервью по базовым темам.

2. **ИИ-отчёт из 13 разделов** (R-01 – R-13) — готовый материал для стратегической сессии, брифа на лендинг и подготовки рекламных сообщений.

3. **PDF-файл** — итоговый документ для архива, отправки клиенту или печати.

4. **Список недостающей информации** — конкретные вопросы для одного короткого уточняющего созвона вместо полного повторного интервью.

5. **Основу для следующих этапов** (вне системы): структура лендинга, офферы, позиционирование, заготовки для рекламных текстов.

### 8.2. Клиент получает

1. **Структурированный процесс** — возможность осмыслить свой проект через guided-опросник.
2. **Подтверждение завершения** — понимание, что информация передана специалисту.
3. **Опционально (через маркетолога):** PDF-отчёт — если маркетолог решит его передать.

Клиент **не** получает автоматический доступ к ИИ-отчёту внутри системы в MVP — только через маркетолога.

### 8.3. Разделы из устава, сознательно сокращённые в MVP

Следующие результаты из полного видения продукта **не генерируются** в первой версии как отдельные полноценные разделы (переносятся в v2):

- готовые **тексты лендинга** (полный copywriting страницы);
- **рекламная стратегия** (каналы, бюджеты, медиаплан);
- **идеи рекламных креативов**;
- **контент-план**;
- **рекомендации по улучшению продукта** (как отдельный глубокий блок).

В MVP вместо них: **структура лендинга**, **офферы**, **позиционирование** и **следующие шаги** — этого достаточно, чтобы маркетолог начал работу без генерации «под ключ» материалов, качество которых сложно гарантировать в сжатые сроки.

## 9. Что отложить на следующие версии

## 10. Структура опросника (минимальный состав блоков)

Фиксированный опросник MVP — **6 блоков**. Точные формулировки вопросов — отдельный документ (`04_DIAGNOSTIC_QUESTIONNAIRE.md`, будет создан на следующем этапе).

| Блок | Тема | Примерное кол-во вопросов | Цель |
|------|------|---------------------------|------|
| B-01 | О проекте и эксперте | 5–7 | Продукт, опыт, экспертиза, текущий этап |
| B-02 | Целевая аудитория | 5–7 | Кто клиент, как живёт, что важно |
| B-03 | Проблема и результат | 5–7 | Боли, желаемый результат, трансформация |
| B-04 | Конкуренты и рынок | 4–6 | Альтернативы, отличия, ценовой контекст |
| B-05 | Цели и метрики | 4–5 | Зачем запуск, KPI, сроки, бюджетные ожидания |
| B-06 | Ресурсы и ограничения | 4–5 | Что уже есть, что мешает, табу, страхи |

**Общее время заполнения клиентом:** 30–60 минут.

## 11. Риски

### 11.1. Продуктовые риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| ИИ генерирует поверхностный или шаблонный анализ | высокая | критическое | жёсткая структура разделов, опора на конкретные ответы клиента, блок «недостающая информация», ручная проверка маркетологом перед передачей клиенту |
| Клиент бросает диагностику на середине | высокая | высокое | автосохранение, понятный прогресс, возможность вернуться, короткие блоки |
| Ожидания первого клиента шире MVP | средняя | высокое | явное согласование scope до начала; PDF и отчёт как «версия 1» |
| Фиксированный опросник не подходит под нишу | средняя | среднее | формулировки вопросов универсальны; ниша указывается при создании проекта |

### 11.2. Операционные риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Сжатые сроки (несколько дней) | высокая | критическое | жёсткий scope MVP, отказ от интеграций и второстепенных разделов |
| Нет информационной архитектуры (`02_*.md` пуст) | факт | среднее | данный документ закрывает пробел; IA оформить отдельно перед разработкой UI |
| Нет Product Vision (`01_PRODUCT_VISION.md` пуст) | факт | среднее | не блокирует MVP; charter достаточен для v0.1 |

### 11.3. Риски качества данных

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Клиент отвечает односложно | высокая | высокое | подсказки к вопросам, примеры хороших ответов, блок недостающей информации |
| Противоречия в ответах клиента | средняя | среднее | ИИ отмечает противоречия в резюме или блоке «следующие шаги» |

### 11.4. Риски доверия и конфиденциальности

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Утечка данных клиента через публичную ссылку | низкая | критическое | длинный уникальный токен, без индексации, HTTPS |
| Клиент делится ссылкой с третьими лицами | низкая | среднее | принять как допустимое поведение в MVP |

## 12. Definition of Done для MVP

MVP готов к передаче первому клиенту, когда выполнены **все** пункты:

- [ ] Маркетолог может создать проект и получить рабочую ссылку
- [ ] Клиент может пройти все 6 блоков диагностики без регистрации
- [ ] Ответы сохраняются и не теряются при повторном входе
- [ ] После завершения диагностики автоматически формируется ИИ-отчёт (разделы R-01 – R-13)
- [ ] Маркетолог видит ответы и отчёт в интерфейсе
- [ ] PDF скачивается и содержит все разделы отчёта
- [ ] Статусы проекта корректно отражают жизненный цикл
- [ ] Проведён end-to-end прогон на реальном или максимально близком к реальному кейсе
- [ ] Маркетолог подтвердил, что отчёт пригоден для работы без повторного базового интервью

## 13. Открытые вопросы (требуют решения до разработки)

Согласно `00_PROJECT_RULES.md`, при недостатке документации — не додумывать, а зафиксировать вопросы:

1. **Бренд в интерфейсе:** ProjectMind или AI Marketing System — что видит клиент на экране диагностики?
2. **Передача PDF клиенту:** маркетолог отправляет вручную или нужна кнопка «Отправить на email клиента» уже в MVP?
3. **Разблокировка диагностики:** достаточно ли переключателя у маркетолога или нужен отдельный flow для клиента?
4. **Юридическое согласие:** нужно ли клиенту принимать политику обработки данных перед началом?
5. **Лимит проектов:** один маркетолог в MVP или закладываем несколько аккаунтов?
6. **Точные формулировки вопросов:** утвердить тексты опросника до начала разработки UI.
